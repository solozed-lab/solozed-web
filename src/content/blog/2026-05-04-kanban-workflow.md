我知道大部分人对「看板」这个词的第一反应是：又是一个项目管理工具？

我先不解释它是什么，先说一个更具体的问题。

**相信你也遇到过：跟 AI 协作一顿操作，中间有好几轮来回，第二天打开对话发现状态丢了**

「我上次那个调研做到哪了？」
「这个结论是谁出的？」
「那个 bug 是已经修了还是还在查？」

不是 AI 不聪明，而是中间的状态没有地方存。对话记录在滚动，session 重启，上下文压缩——你做过的每一步，中间有什么产出，下一步该做什么，全靠你自己记。

我之前靠一套 `tasks/` 目录结构来解决这个问题。每个任务建一个文件夹，里面放 `spec.md`、`agent-log.md`、`inbox/`、`outputs/`。能跑通，但没有跨 session 的追踪能力。第二天打开，我得先读一遍 `agent-log.md` 才能想起「哦对，做到这一步了」。

于是我试了另一个方案。

Hermes 最近支持了看板功能。简单说就是每个 task 有个状态，做完自动标记，父 task 完成子 task 自动就绪。不需要人盯着。

这些都不重要。重要的是它帮我解决了一个很烦人的问题。

**我不需要再手动追踪状态了。**

区别在哪？之前我说「帮我调研一下 A」，AI 去做，做完告诉我。我记着「哦 A 做完了」，然后说「那基于 A 的结论去做 B」。每一步的**状态**在我脑子里，在我手动翻对话记录的过程里。

用了看板之后：我说「帮我调研一下 A」，AI 创建 task，完成后自动标记 `done`。task B 的 parent 是 task A，A 的 `done` 了 B 自动变成 `ready`。我不需要任何人为的「记得 A 做完了吗」。

**看板就是状态。不是帮我记录状态，它就是状态本身。**

我当天用 6 个 Kanban task 跑了一遍自己的日常工作。创建、指派、完成、链式依赖，全程没有任何一步需要我手动去「检查当前到了哪一步」。我只需要打开看板，`done` 就是 `done`，`ready` 就是 `ready`——不用猜。

实际体验差别很大。之前协作时每隔一段时间就要中断一下思路去确认**当前状态**——现在不用了，看板告诉你就够了。

**你可能会说，这不就是个 todolist 吗**

不是。区别在于 todolist 是给自己看的，而看板是给 AI 协作中的**状态**一个地方住。每次 AI 完成一个步骤，它自己会把 task 标记为 `done`，写入 `summary` 和 `metadata`。下次 AI 启动时，先读当前 task 的状态，就知道从哪接上。在这个过程中，AI 不需要问「做到哪了」，人不需要说「做到调研那一步了」。看板就是那个共同的状态容器。

我也不是彻底抛弃了之前的 `tasks/` 目录。`tasks/` 仍然是我写 spec 和放交付物的地方，看板负责排队和追踪。两者不重复：看板 body 引用 `tasks/spec.md` 的路径，`kanban_complete` 的 summary 引用 `tasks/outputs/` 的路径。一个负责持久化队列，一个负责工作区。

工具好不好用，不看它有多少功能。看它能不能让你少记一件事。

少记一件事，省出来的不光是时间。是协作时脑子不用频繁中断、不用反复确认**刚才做到哪了**的那种从容。

AI 时代，值得更加从容。
