和 Hermes 协作效率提升不少，但也有复杂的任务效果一直不太好。

最常用的工作方式就是：我下指令，Hermes 执行。聊着聊着，上下文满了，然后压缩，之前的内容丢一部分，精度和准确度也跟着下降。

很多时候不是 Hermes 不够强，是我不够会管它。

因为没有决策者，每个AI都按自己的理解执行，最后结果散成一片。

后来我想明白了——需要一个真正做决策的人，不能让 AI 自己协调。

所以我决定试试：**我当甲方，Hermes 当 CEO，让它来统筹，结果会怎样？**

---

## 我之前走了很多弯路

之前的做法简单粗暴——把需求扔给 AI，让它自己干。

**结果呢？**

**V1：直接让 AI 干。**

AI 自由发挥，写出来的东西跟我想的完全不是一回事。我开始往 prompt 里加约束。

**V2：加 prompt。**

prompt 越来越长，AI 开始偷懒。你说什么它应什么，但实际做的时候自己变通。

**V3：我引入了角色分工。**

researcher、writer、editor 三个 AI 各管一摊。结果呢？没有决策者，三个 AI 各干各的，打起来了。我成了传话筒。

---

## 后来我想清楚了

需要一个真正做决策的人。

**V4：CEO 模式。**

主 agent 做 CEO，我当甲方。AI 做执行者，我做判断。

但光有 CEO 不够。AI 不知道什么情况下该做什么事，需要一套明确的规则。

**V5：skill 体系。**

把「什么人负责什么事」固化下来。需要什么，调用什么。

比如：

```Markdown
CEO（主 agent）
├── 产品经理（implementer）  — 功能实现、代码编写
├── 架构师（architect）      — 技术方案设计
├── bug 诊断师（investigator）— 定位问题根因
├── UI/UX 评审（designer）  — 视觉与交互规范
└── 智囊团（advisor）        — 复杂决策多角度评审
```
实际用的时候按需组合，不是一股脑全上。

---

## 我的团队现在是这样运作的

四场景协作：

| 场景              | 触发条件                  | 交付物              |
|-------------------|---------------------------|---------------------|
| 直接做            | 2分钟内 + 改坏能还原      | 无正式交付          |
| 单Agent+验收      | 单文件 + 局部影响         | 修改记录            |
| 团队串行          | 多步骤 + 有依赖           | 轻量SPEC + 交接记录 |
| CEO写SPEC         | 多文件/全局影响           | 完整SPEC.md         |


但有几条铁律，CEO 必须遵守：

```Markdown
CEO 三条红线：
1. CEO 不写代码（决策归甲方，执行归 subagent）
2. Bug 必须先诊断（不能直接让 AI 猜着改）
3. 全文件 grep 验证（改了一处要全文件检查一致性）
```

## 这一天我验证了什么

回到 Solace AI 这个项目。

前两天我对 Hermes 说：「你是 CEO，分派你的智囊团、产品经理、UI 设计团队，对整个产品做一份全景设计——详细到每个页面有什么模块、每个按钮有什么用、每个操作会导致什么结果。」

这是一个很大的任务。

如果放在以前，我会在一个会话里自己干，边想边写，边写边改，最后自己都说不清想要什么了。

但这次不一样。

我让 CEO 先列计划。CEO 说：「这个任务太复杂，我先让智囊团开会对齐，分工，然后每个 subagent 只处理一个最小模块，做完及时汇总，再派下一个。」

结果呢？

CEO 分派了多个 subagent 并行处理——产品经理梳理功能结构，UI 设计评审视觉规范，智囊团评估技术可行性。每个 agent 只做一件事，做完汇报，CEO 汇总。

整个过程，我只说了需求，然后验收结果。

那天我还顺带验证了另一件事：Markdown 项目需要一次 code review。我同样是让 CEO 分派团队处理，结果比我自己盯着改要好得多。

**不是因为我干不了，是因为我一个人干，不如一个团队各司其职。**

就像你说的：「多 agent 分工处理复杂任务，效率和准确率都比一个会话单独干要强。」

## 一天结束的时候

回头看那天的结果：

- 一份完整的产品全景设计文档，细节到每个交互逻辑
- 一份 Code Review 报告，记录了所有需要关注的点
- 还有几个 subagent 执行任务时的日志，可以追溯每一步

这不是我一个人干的。

我下达任务，主agent规划执行，我们一起完成的。

---

## 我学到了什么

**第一，AI 不是帮你偷懒，是帮你「看见」。**

很多人觉得 AI 就是个高级搜索引擎。不是的。它能帮你看到你之前看不到的东西——你的盲点、你的流程问题、你的机会。

**第二，AI 给你的是「框架」，不是「答案」。**

它说「写一封邮件」，你还是要自己填内容。但那个框架已经帮你省了 80% 的力气。

**第三，最重要的变化是：终于不用一个人扛了。**

这种感觉听起来很虚，但其实是最实在的。

## 我们不要被自己既定的习惯思想所裹挟

这套方法不是我想出来的。

Andrew Ng (吴恩达) 曾提出了构建 AI 智能体工作流（Agentic Workflow）的四种核心设计模式，其中一种被称为「多智能体协作 (Multi-agent collaboration)」。

Claude Code 背后也是这个逻辑。它通过子智能体和智能体群组机制来协调多个 agent，同时可以利用 CLAUDE.md 文件为每个 agent 定义基础行为规范。

社区里把这个叫做「乐队指挥」——你指挥，多个 AI 各演各的。

我的做法也一样：**人建立秩序，AI 执行。**

协作准则比工具重要。
